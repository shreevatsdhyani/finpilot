FinPilot AI Frontend UI Specification and Delivery Roadmap

Product Positioning
FinPilot AI should feel like a premium personal finance workstation rather than a generic dashboard. The interface should balance trust, clarity, and analytical depth. It should look suitable for salary earners who want actionable planning, local privacy, transparent recommendations, and a strong sense of control over their financial data.

Primary Market Assumption
- Region: India-first
- Currency: INR by default
- Locale: en-IN
- Product tone: calm, professional, transparent, practical

Core Experience Principles
1. Trust first
   Every important number should hint at where it came from, when it was last updated, and whether it is observed, estimated, or simulated.

2. One clear next step per screen
   Each page should have one dominant CTA. Secondary actions can exist, but there should always be a clear next move.

3. Local-first privacy
   The app should emphasize export, retention control, and deletion of local data. It should never feel like a surveillance or sharing product.

4. Explainability by default
   Recommendations, summaries, assistant answers, and forecasts should expose evidence, assumptions, confidence, and audit paths.

5. Dense but readable
   The UI should support analytical users. That means more information can be shown, but hierarchy, spacing, grouping, and labels must keep it readable.

6. Great empty states
   New users will often start with no data. Empty states should guide setup and make the system feel intentionally incomplete rather than broken.


Visual Direction
Theme Direction
- Primary visual language: white, black, charcoal, and green
- Secondary support accents: muted mint, steel gray, and soft emerald
- Charts should use a restrained analytic palette with one highlight green and neutral comparison tones

Light Theme
- Background: warm white or very pale gray-green
- Surface: white with subtle tint
- Primary: deep green
- Accent: mint or cool green-gray
- Text primary: near-black
- Text secondary: slate/gray
- Border: pale gray-green

Dark Theme
- Background: near-black or forest-charcoal
- Surface: charcoal/graphite
- Primary: rich green
- Accent: mint-teal glow used sparingly
- Text primary: off-white
- Text secondary: muted gray-green
- Border: dark olive-gray

Visual Behavior
- Use gradients, glass-like soft panels, and atmospheric background layers without making the app flashy
- Motion should be subtle: card hover lift, chart reveal, panel fade-in, skeleton shimmer
- Border radius should be modern and slightly soft, not overly rounded everywhere
- Shadows should be restrained and layered to create depth

Typography
- Modern sans stack is acceptable
- Headings should feel intentional and financial, not playful
- Large KPI numbers should be highly legible and aligned for scanning

Charts
- Use Recharts throughout
- Charts must include:
  - direct labels or strong legends
  - units on axes/tooltips
  - readable contrast in both themes
  - explicit assumptions for forecasts/projections
- Recommended chart types:
  - line chart for actual vs forecast trends
  - stacked bars for cashflow breakdown
  - donut or radial charts for allocation and completion
  - comparison bars for scenarios
  - area charts for projection bands and confidence ranges

Cross-App Reusable UI Patterns
1. Trust strip
   A compact component that can show:
   - Updated date
   - Source type
   - Confidence
   - Audit link

2. Empty state card
   Each empty state should contain:
   - a direct explanation of why the section is empty
   - the next action CTA
   - optional secondary guidance

3. Explainability panel
   Reusable panel showing:
   - Why this exists
   - Inputs used
   - Assumptions
   - Confidence
   - Sources / Evidence
   - Audit link

4. KPI card
   Structure:
   - label
   - value with units
   - change indicator or confidence chip
   - “Edit inputs” or “View detail” action

5. Audit entry reference
   Compact inline component used in Assistant, Recommendations, News summaries, Forecasts

6. Skeleton states
   Needed for:
   - page headers
   - KPI cards
   - charts
   - tables
   - assistant messages

7. Error states
   Friendly but serious finance-oriented error states:
   - failed to load data
   - failed to generate forecast
   - OCR extraction issue
   - assistant retrieval unavailable


Page Specifications

1. Dashboard
Purpose
- Show current financial position
- Show most important next step
- Make the system feel intelligent and alive

Required Sections
- KPI strip:
  - Monthly income
  - Monthly expenses
  - Net savings
  - Safe-to-invest
  - Active goals
- Primary insight card:
  - “Your next best action”
  - one CTA only
- Income vs expenses chart, 6 months
- Expense breakdown chart
- Risk months panel
- Goals progress section
- Freshness and trust row

Empty State Logic
- No salary docs: CTA to upload salary slip
- Salary exists but no expenses: CTA to add monthly expenses
- Salary + expenses exist but no forecast: CTA to generate 6-month forecast
- Forecast exists but no goals: CTA to add first goal

Make It Work
- Data source:
  - GET /v1/dashboard/summary
  - GET salary list if richer breakdown needed
  - GET expenses if chart data needed
  - GET goals if progress cards needed
  - GET forecast list if trend/risk chart needed
- Current backend summary supports empty-state metrics but richer dashboard will likely need page-level aggregation from multiple endpoints.

Filled State Experience
- User sees salary-informed income
- Expense categories appear visually
- Forecast risk months are called out with reason tags
- Goals show projected completion timing

Data Freshness and Trust Language
Every analytical page should expose at least one of:
- Salary updated: dd mmm yyyy
- Expenses updated: dd mmm yyyy
- Forecast generated: dd mmm yyyy
- Recommendation based on data up to: dd mmm yyyy
- Scenario active: name
- Confidence: high / medium / low

Avoid language like exact prediction. Prefer:
- estimated
- projected
- based on current inputs
- confidence medium
- assumptions editable

How to Make Each Page Feel Filled Even Early
1. Use derived summaries from existing API data
   If backend lacks a dedicated summary endpoint for a page, derive aggregates in the frontend from list endpoints.

2. Design rich empty states, not blank surfaces
   Each empty state should still teach the user what the system will show once data exists.

3. Use trust panels as meaningful content
   Even before all charts are populated, freshness, assumptions, privacy, and explanation panels make the product feel complete.

4. Stage chart density progressively
   Start with one high-value chart per page, then add secondary charts once data model support is stable.

5. Keep the context bar informative
   Even a partially configured app should show what is missing and what is active.

Success Criteria for Frontend Redesign
- The app looks premium in both light and dark mode
- Each page has a clear primary action
- Data origin and freshness are visible on key surfaces
- Empty states feel intentional and helpful
- Assistant and recommendations expose evidence and audit links
- Charts are readable and attractive
- The product feels like a serious finance tool, not a template admin dashboard


Final Note
The best delivery strategy is not to beautify page by page independently. Build the shell, tokens, trust patterns, chart language, and empty-state system first. After that, each page becomes a composition problem rather than a styling problem. That is what will make the entire app feel coherent and high-end rather than partly redesigned.
